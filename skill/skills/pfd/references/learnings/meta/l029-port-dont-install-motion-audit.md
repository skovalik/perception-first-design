# Learning 29: Port-Don't-Install Motion Audit — Library Components Ship Embedded Motion Postures the Visual Designer Doesn't See

**Insight:** Calibration testing across the campaign's library-integration work — 15 reactbits / 21st-magic library components integrated across 11 episodes per `LIBRARY-OPPORTUNITIES.md` — surfaced an engineering-discipline primitive PFD has not previously named: library components ship with embedded motion postures the visual designer doesn't see, and library adoption without active port-time audit silently injects motion debt that violates PFD's L2 fluency guardrails and #14 operational-ethics commitments. The campaign's discipline was *port to vanilla HTML/CSS/JS rather than install as runtime dependency*, and the discipline was load-bearing in a way that wouldn't have been visible without it. The mechanism: library components are authored by their original creators with default motion behaviors that often fail PFD's `prefers-reduced-motion` guardrails by default — `GridDistortion` ships with cursor-reactive distortion that runs regardless of user motion preference; `Particles.moveParticlesOnHover` ships with cursor-tracking particle motion that the visual API surface hides from the integrator; many hero-frame components ship with auto-loop animations that don't respect reduced-motion media queries. Installing these as npm runtime dependencies inherits the embedded postures silently — the visual API call site shows nothing about the motion behavior, and the motion debt only surfaces when a real user with `prefers-reduced-motion: reduce` set encounters the surface in production. The campaign's port discipline forced *every* component to re-validate its motion posture at port time, surfacing the violations before they shipped. The campaign's `LIBRARY-OPPORTUNITIES.md` includes an explicit "Excluded components (PFD violations)" section catalogueing the components that failed the port-time audit and were therefore not integrated despite visual fitness — the catalogue is the documented evidence that without port-time audit, these components would have shipped invisibly. The promoted spec is two-rule: **(1) port, don't install** — when integrating a library component into a PFD-compliant surface, port the component's source to the project's vanilla stack rather than installing the library as a runtime dependency, with the port serving as a forced read-and-audit pass on every component file; **(2) audit motion posture at port time** — for every ported component, explicitly check `prefers-reduced-motion` compliance, cursor-reactive behaviors, auto-loop animations, scroll-driven effects, and any other motion not specified in the component's documented API; flag components that fail PFD guardrails as excluded rather than ported with patches. The discipline is engineering-side, not audit-side — it operates at the moment of library adoption, not during user-facing audit. Generalizes beyond the campaign's reactbits / 21st-magic context to any library adoption where the consumer cares about constraints the library author may not have prioritized: Tailwind component libraries, shadcn/ui registry, Material UI / MUI, Chakra UI, Mantine, NextUI, daisyUI — every component ecosystem ships hidden motion (and hidden accessibility, hidden contrast, hidden cognitive-load) defaults that surface only at port-time read. The same discipline extends to non-motion concerns: a port-time audit on Tailwind class compositions for AA-contrast (Learning #24); on shadcn for keyboard-density (Learning #13); on any library for the Visual Channel allocation it implicitly makes (Learning #11). The cost-benefit calculation favors port: the cost is engineering time at integration (one-time, ~1–4 hours per component); the benefit is preventing the invisible-debt class of violations that compound across every page using the component, multiplied by every user with the relevant accessibility preference. The honest scope qualifier — the campaign's evidence is 15 components across 11 episodes, all from a small set of source libraries (reactbits, 21st-magic); generalization to broader library ecosystems (Tailwind, shadcn, Material) is predicted by mechanism (every library ships hidden defaults) but not measured in this calibration sample. Validate on 2–3 more library ecosystems before treating the discipline as fully established beyond the source set. Practical heuristic for any team adopting a third-party component library: budget engineering time for the port-and-audit pass at integration; if the budget is unavailable, the alternative is *don't adopt the library* — installing without auditing is the failure mode this learning names.
