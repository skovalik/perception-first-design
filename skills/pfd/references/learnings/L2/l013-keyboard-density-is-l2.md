# Learning 13: Keyboard Density in Expert Workspaces Is L2, Not L4

**Insight:** When multiple RL personas flagged "too many keyboard shortcuts" in Phase 5 (10 competitive patterns, 4+ new shortcuts), the initial diagnosis was L4 choice overload. After L4 was updated with the Scheibehenne (2010) context qualifier — choice overload is near-zero in expert workspaces — the problem reframed: the issue is L2 Processing Fluency (users cannot predict what a keypress will do across cognitive modes), not L4 (too many options). The fix is identical (keyboard mode indicator showing Navigate/Edit/Terminal), but the layer assignment matters because it changes what you test. L4 testing asks "are there too many options?" (wrong question for expert tools). L2 testing asks "can users predict the result of their next action?" (right question). In expert workspace contexts, shortcut density is a fluency/predictability problem, not a choice overload problem. Don't reduce expert affordances; signal which mode they operate in.
